## 一、系統架構總覽

```
Python (YOLO + ST-GCN)
        │ POST /yolo_result_v2
        ▼
┌─────────────────────────────────────────────────────┐
│  第1層：核心資料流                                      │
│  接收 → 行為統計累積器 → P1 / P2 面板 → 分析引擎          │
└─────────────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────────────┐
│  第2層：行為分析引擎                                     │
│  行為分布 → 節律分析 → 偏差分析 → 健康預警引擎               │
│       ↓         ↓          ↓                        │
│     P3        P4          P5       Discord 通知      │
└─────────────────────────────────────────────────────┘
        │（baseline_days 變更 / 排除日期切換）
        ▼
┌─────────────────────────────────────────────────────┐
│  第3層：基線引擎                                        │
│  個體化基線計算器（mean / std / median）                  │
└─────────────────────────────────────────────────────┘
        ↑
┌─────────────────────────────────────────────────────┐
│  定時任務                                              │
│  每日午夜彙整 → 判斷是否重算 → 觸發基線                    │
│  每小時偏差快照（CSV）                                   │
│  系統啟動跨日清除                                        │
└─────────────────────────────────────────────────────┘
```

Node-RED flow 共 **4 個 Tab、55 個節點**。

## 二、Tab 說明

### Tab 1：第1層 核心資料流

| 節點 | 類型 | 說明 |
|------|------|------|
| 接收Python數據 | HTTP In | `POST /yolo_result_v2` 接收 Python 端推送的即時偵測結果 |
| 200 OK | HTTP Response | 立即回覆 200，不阻塞 Python |
| 行為統計累積器 | Function | 解析 `today_stats`、`current`，寫入 `v2_today`（file store）；每5分鐘輸出一次 CSV；產出 4 路輸出（→P1、→P2、→分析引擎、→CSV） |
| P1 即時監控面板 | UI Template | 顯示當前行為 emoji、行為 bar、活動力/健康分/風險分 |
| P2 今日統計面板 | UI Template | 5種行為的次數/時間卡片、每小時活動分布長條圖 |

**每日統計欄位**（`v2_today`）：

```
walk, walk_time / lick, lick_time / scratch, scratch_time / shake, shake_time / stop, stop_time
active_time, rest_time, not_detected_time, monitoring_seconds
transition_matrix, hourly_distribution
```

---

### Tab 2：第2層 行為分析引擎

資料從分析引擎入口流入，依序經過 4 個計算節點再輸出至各 UI 面板。

#### 計算流水線

```
行為分布計算 → 行為節律分析 → 偏差分析引擎 → 健康預警引擎
```

**① 行為分布計算**
- 計算 5 種行為各佔總時間的百分比
- 輸出 `distribution` 物件

**② 行為節律分析**
- 將 24h 分成 4 個時段：`00-06 / 06-12 / 12-18 / 18-24`
- 計算各時段的主導行為（dominant）及各行為佔比
- 輸出 `rhythm` 物件

**③ 偏差分析引擎**
- 與 `v2_baseline`（個體化基線）比較
- 計算各指標相對於基線 mean 的百分比偏差及 σ 值
- 無基線時 `deviation_available = false`

**④ 健康預警引擎**（核心）

風險分公式：

```
riskScore = dScore×30% + fScore×35% + rScore×20% + tScore×15%
```

| 分量 | 說明 | 不需基線 |
|------|------|----------|
| **dScore**（行為占比偏離，0~100） | 今日各行為比例是否超過固定門檻 | ✅ |
| **fScore**（行為頻率偏離，0~100） | 與個體基線相比各行為是否異常偏高 | ❌ 需基線 |
| **rScore**（行為節律偏離，0~100） | 深夜搔抓/甩頭、白天活動量不足 | ✅ |
| **tScore**（行為轉移偏離，0~100） | lick↔scratch 循環、shake→shake 連續 | ✅ |

風險等級分界：

| 分數 | 等級 | 顏色 |
|------|------|------|
| 0~19 | Normal | 綠 |
| 20~44 | Attention | 橘黃 |
| 45~69 | Warning | 橘紅 |
| 70~100 | High Risk | 紅 |

**預設行為門檻**（可在飼主設定調整）：

| 設定 | 預設 |
|------|------|
| 舔舐占比上限 | 20% |
| 搔抓占比上限 | 15% |
| 甩頭占比上限 | 10% |
| 靜止占比上限 | 55% |
| lick↔scratch 警戒次數 | 10 次 |

#### 使用者設定管理器（`ff00000000000010`）

接收來自 P4 面板的 4 種 action：

| action | 說明 |
|--------|------|
| `save_settings` | 驗證並寫入 `v2_user_settings`（file store）；baseline_days 有變→立即觸發基線重算 |
| `save_event_tag` | 寫入 `v2_event_tags`（file store），同一天 upsert |
| `toggle_exclude_date` | 切換 `v2_excluded_dates` 排除清單，立即觸發基線重算 |
| `test_discord` | 發送測試訊息到 Discord Webhook |

**Output 1** → 觸發基線重算
**Output 2** → Discord HTTP Request

---

#### Discord 風險通知

- 只在 `健康預警引擎` output 2（需通知）才觸發
- dedup key = `risk.level + '|' + alerts 類型串`，同樣的警報不重複推播
- 通知靈敏度（`notify_level`）由飼主設定控制：

| 等級 | 觸發條件 |
|------|----------|
| 1（低） | High Risk（分數 ≥ 70）才推播 |
| 2（中，預設） | Warning 以上（分數 ≥ 45）推播 |
| 3（高） | Attention 以上（分數 ≥ 20）即推播 |

### Tab 3：第3層 基線引擎

**個體化基線計算器**（`66a22e0ec663d07d`）

篩選邏輯（三層過濾）：

```
v2_daily_history
   ↓ 層1：monitoring_seconds >= MIN_DAILY_SEC (3600 秒)
   ↓ 層2：手動排除（v2_excluded_dates）
   ↓ 層3：取最近 30 天
   ↓ 計算 mean / std / median
→ 寫入 v2_baseline（file store）
```

| 設定 | 說明 | 預設 |
|------|------|------|
| `MIN_BASELINE_DAYS` | 從 `v2_user_settings.baseline_days` 讀取 | 7 天 |
| `MIN_DAILY_SEC` | 每天最少監控秒數 | 3600（1小時） |
| `REQUIRED_PERIODS` | 限定哪些時段必須有資料 | `[]`（不限） |

基線指標（各指標均計算 mean / std / median）：

`lick_count / lick_time / scratch_count / scratch_time / shake_count / shake_time / walk_count / walk_time / stop_time / stop_count`

---

### Tab 4：定時任務

| 節點 | 觸發時機 | 說明 |
|------|----------|------|
| 每日午夜觸發 | 00:00 每天 | 觸發「每日統計彙整」 |
| 每小時快照 | 每 3600 秒 | 輸出偏差快照至 `deviation_log.csv` |
| 系統啟動檢查 | 啟動後 2 秒 | 偵測跨日並清除舊的 `v2_today` |
| 手動觸發基線計算 | 手動 | 直接觸發基線計算器 |
| 手動觸發每日彙整 | 手動 | 手動將今日資料存入歷史 |
| 手動觸發全部重置 | 手動 | 清除所有 v2_today / v2_hourly / v2_matrix |

**每日統計彙整**（`c5ce881398ee6376`）：
- 從 `v2_today` 讀取累積資料
- 計算 4 個時段的行為分布（`periods`）
- 同一天 upsert 去重後 push 進 `v2_daily_history`（最多保留 90 天）
- 若 `monitoring_seconds < 1800` 秒則跳過記錄
- 重置 `v2_today` 為新一天的空資料

## 三、File Store 全域變數

| Key | 說明 | 讀取節點 |
|-----|------|----------|
| `v2_today` | 今日行為累積（5種行為×count/time） | 行為統計累積器、每日統計彙整 |
| `v2_hourly` | 今日每小時分布 `{HH: {walk,lick,...}}` | 行為節律分析、每日統計彙整 |
| `v2_transition_matrix` | 行為轉移矩陣 `{A->B: count}` | 健康預警引擎 |
| `v2_baseline` | 個體化基線（mean/std/median） | 偏差分析引擎、P4 面板 |
| `v2_daily_history` | 已完成天的歷史記錄（最多 90 天） | 個體化基線計算器、P4 面板 |
| `v2_user_settings` | 飼主設定（閾值、通知、貓咪資料等） | 健康預警引擎、基線計算器 |
| `v2_event_tags` | 事件標記記錄（最多 90 筆） | 健康預警引擎→P4 面板 |
| `v2_excluded_dates` | 手動排除的基線日期清單 | 個體化基線計算器、P4 面板 |
| `v2_deviation` | 最新偏差分析結果 | 小時偏差快照 |
| `v2_risk` | 最新風險分計算結果 | 小時偏差快照 |

| Key | 標籤 |
|-----|------|
| `vet` | 🏥 帶去看醫生 |

| `stress` | 😿 情緒緊張 |

---

### 📋 歷史資料管理（摺疊區塊）

顯示最近 14 天的歷史記錄，每筆顯示：
- 日期
- 監控時數（h）
- 基線資格標籤：

| 標籤 | 條件 | 說明 |
|------|------|------|
| ✓ 計入（綠） | monitoring_seconds ≥ 3600 且未被排除 | 納入基線計算 |
| ⚠ 太短（灰） | monitoring_seconds < 3600 | 自動不計入 |
| （排除按鈕顯示）↩ 取消排除 / ✕ 排除此天 | 僅有效天才顯示 | 手動切換排除 |
| 不計入（淡字） | 太短時取代按鈕 | 不可操作 |

- 按下排除/取消後立即重算基線
- 3 秒內 `_excludePending` 旗標防止伺服器舊訊息覆蓋 UI 狀態

## 五、資料流時序圖

```
Python 每秒推送
      │
      ▼
行為統計累積器（累積 v2_today）
      │
      ├──→ P1 即時監控（每次更新）
      ├──→ P2 今日統計（每次更新）
      └──→ 分析引擎鏈（每次更新）
                │
                ├──→ P3 行為分析
                ├──→ P4 個體基線（含 v2_daily_history、v2_baseline）
                └──→ P5 健康預警（含 Discord 通知）

每日 00:00
      │
      ▼
每日統計彙整 → v2_daily_history (push + upsert去重)
      │
      ▼
判斷是否重算基線（每 baseline_days×2 天強制重算一次）
      │（若需重算）
      ▼
個體化基線計算器 → v2_baseline

每小時
      │
      ▼
小時偏差快照 → deviation_log.csv
```

## 八、最近修復紀錄（2026-06-16 ~ 06-17）

### Bug 修復

| Bug | 根本原因 | 修復方式 |
|-----|----------|----------|
| 排除基線按鈕按了沒反應 | 使用者設定管理器 JS 末尾多一個 `}`，語法錯誤整個 function 無法執行 | 移除多餘 `}`，大括號平衡 18/18 |
| 飼主設定儲存後打開還原預設 | 同上（SM 無法執行）+ 引擎週期性訊息在 3 秒內覆蓋 UI 值 | 修正語法 + 加入 `_settingsSaving` 旗標（3秒） |
| 排除按鈕被舊訊息覆蓋 | `$watch` 在 toggle 後立即以伺服器回傳覆蓋本地狀態 | 加入 `_excludePending` 旗標（3秒） |
| 同一天歷史記錄重複兩筆 | `history.push(rec)` 無去重 | 改為 `filter(h => h.date !== date)` 後再 push（upsert） |

## 九、重新部署步驟

每次修改 `cat_health_v2_flow.json` 後需手動匯入：

1. Node-RED 右上角選單 → **Import**
2. 選擇 `cat_health_v2_flow.json`
3. 選「**Replace existing flow**」
4. 按 **Deploy（全部部署）**

> file store 資料（`v2_daily_history` 等）在重新部署後會保留，不會被清除。
