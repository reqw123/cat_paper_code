# 舔舐行為二階段分析模組說明

## 一、目的與需求背景

### 問題描述

ST-GCN 動作分類器輸出的行為標籤（如 `lick`）僅能判斷「貓咪正在舔舐」，無法進一步區分舔舐的**目標身體部位**（舔身體、舔前腳、舔後腳等）。然而在行為分析或醫學評估場景中，舔舐部位的差異具有重要意義：

- 持續舔舐特定肢體可能代表傷口、皮膚病或寄生蟲
- 舔舐頻率與分佈可作為焦慮或疼痛行為的量化指標
- 頭部朝向與接觸區域的組合判斷能過濾視角干擾造成的誤判

### 設計思路：二階段處理

```
第一階段：ST-GCN 行為分類
  ↓ 輸出 lick 標籤
第二階段：lick_stage 精細化分析
  ├── 建立鼻部梯形接觸區域（Nose Trapezoid）
  ├── 計算身體各區域目標多邊形（Body / FL / FR / HL / HR）
  ├── 測試鼻尖是否落入接觸區域
  ├── 推斷頭部朝向（FRONT / LEFT / RIGHT / DOWN / UNKNOWN）
  └── 輸出結構化 LickResult（含區域統計、朝向、穩定度）
```

## 二、模組架構與責任劃分

```
plugins/lick_stage/
├── manager.py          ← 對外公開介面（Plugin Facade）
├── analyzer.py         ← 逐幀協調器（Stateful Orchestrator）
├── config.py           ← 所有可調參數集中管理
├── models.py           ← 資料結構定義（LickResult、ZoneStats）
├── contact_regions.py  ← 幾何計算：梯形建構 + 區域交叉判斷
├── head_direction.py   ← 頭部朝向推論（含平滑與前視守衛）
├── ear_distance.py     ← 耳距計算與身體比例估算
└── overlay.py          ← 視覺化覆蓋層繪製（OpenCV）
```

| 檔案 | 職責 | 是否有狀態 |
|------|------|-----------|
| `manager.py` | 接收原始幀與骨架資料，統一呼叫各子模組，發布 Node-RED | 否（委派給 analyzer） |
| `analyzer.py` | 維護 EMA 平滑狀態、區域統計累計、歷史佇列 | **是** |
| `config.py` | 定義梯形幾何、門檻值、關節索引等所有超參數 | 否（靜態資料類別） |
| `models.py` | 定義 `LickResult` 輸出結構與 JSON 序列化 | 否（純資料） |
| `contact_regions.py` | 依鼻尖位置動態建構梯形，並測試鼻點落入哪個區域 | 否（純函式） |
| `head_direction.py` | 以耳朵相對位置推算頭部朝向，包含多數決平滑 | 否（純函式） |
| `ear_distance.py` | 計算雙耳像素距離，並以軀幹長正規化為無單位比例 | 否（純函式） |
| `overlay.py` | 將梯形、身體橢圓、肢體圓、接觸標記繪製到畫面上 | 否（副作用函式） |

## 三、各子模組詳細說明

### 3.1 `manager.py` — 對外公開介面

#### 類別：`LickStagePlugin`

系統其他元件只需與此類別互動，內部實作細節完全封裝。

```python
class LickStagePlugin:
    def __init__(self, config: LickConfig, publisher=None): ...
    def process(self, frame, keypoints, frame_idx, fps, show_overlay) -> LickResult: ...
    def reset(self): ...
```

| 方法 | 功能說明 |
|------|---------|
| `__init__` | 建立 `LickAnalyzer` 與可選的 `NodeRedPublisher`（MQTT/HTTP 推送） |
| `process` | 接受單幀影像與關節座標陣列，回傳 `LickResult`；同時觸發 Node-RED 發布 |
| `reset` | 清除 analyzer 內部所有統計狀態（重新開始計數） |

---

### 3.2 `analyzer.py` — 逐幀協調器

#### 類別：`LickAnalyzer`

持有所有跨幀狀態，每幀呼叫一次 `update()`，依序執行以下流程：

```
update(frame, keypoints, frame_idx, fps, show_overlay)
  │
  ├─ 1. EMA 關鍵點平滑
  │     new_kp = alpha * current_kp + (1-alpha) * prev_kp
  │
  ├─ 2. 提取關鍵點座標（鼻、左耳、右耳、頸、體中心、前後腿）
  │
  ├─ 3. compute_geometry()  → 建立梯形與各身體目標區域
  │
  ├─ 4. find_nearest_zone() → 判斷接觸區域
  │
  ├─ 5. compute_head_ear_angle() → 計算耳朵夾角方向向量
  ├─ 6. compute_ear_distance()   → 計算雙耳距離與正規化比例
  ├─ 7. infer_face_state_*()     → 推論頭部朝向狀態
  ├─ 8. check_front_view_guard() → 前視守衛（雙耳同寬→排除誤判）
  ├─ 9. smooth_state()           → 30 幀多數決平滑
  │
  ├─ 10. 更新區域統計（hit 次數、累計時間）
  │
  ├─ 11. draw_all_overlays()    → 繪製視覺化（可選）
  │
  └─ 12. 回傳 LickResult
```

**重要內部狀態：**

| 屬性 | 說明 |
|------|------|
| `_prev_kps` | 上一幀平滑後的關節座標（EMA 基準） |
| `_zone_stats` | 各區域的 `ZoneStats`（hit 次數、累計秒數） |
| `_state_history` | 長度 30 的朝向標籤 deque（用於多數決平滑） |
| `_frame_count` | 已處理幀數計數器 |

### 3.3 `config.py` — 參數集中管理

#### 類別：`LickConfig`（dataclass）

所有閾值與幾何參數均在此定義，修改時只需改這一個檔案。

**關節索引（依 YOLO-Pose 17 點格式）：**

| 參數 | 值 | 對應部位 |
|------|----|---------|
| `IDX_NOSE` | 0 | 鼻尖 |
| `IDX_LEFT_EAR` | 3 | 左耳 |
| `IDX_RIGHT_EAR` | 4 | 右耳 |
| `IDX_NECK` | 5（或頸部估算） | 頸部 |
| `IDX_BODY` | 11 | 體中心（臀部中點） |
| `IDX_FL/FR/HL/HR` | 5/6/11/12 | 前後左右腿 |

**梯形幾何參數：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `NOSE_TRAP_THICKNESS_CM` | 2.2 | 梯形沿鼻→體方向的深度（公分） |
| `NOSE_TRAP_TOP_W_RATIO` | 0.10 | 梯形頂邊寬度（相對於體長比例） |
| `NOSE_TRAP_BOT_W_RATIO` | 0.20 | 梯形底邊寬度（相對於體長比例） |
| `NOSE_TRAP_THICKNESS_PX_FALLBACK` | 22 | 無法估算公分時的像素後備值 |

**頭部朝向門檻：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `FRONT_VIEW_EAR_RATIO_MIN` | 0.85 | 雙耳距比例下限（觸發前視守衛） |
| `FRONT_VIEW_EAR_RATIO_MAX` | 1.15 | 雙耳距比例上限（觸發前視守衛） |
| `STATE_SMOOTH_WINDOW` | 30 | 多數決平滑使用的幀數視窗 |

**EMA 平滑：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `KP_EMA_ALPHA` | 0.7 | 關節位置時間平滑係數（0~1，越小越平滑） |

### 3.4 `models.py` — 資料結構定義

#### `ZoneStats`（dataclass）

累計單一接觸區域的統計資料。

| 欄位 | 型別 | 說明 |
|------|------|------|
| `hits` | `int` | 接觸事件次數（從「無接觸」變為「接觸」計一次） |
| `time_sec` | `float` | 累計接觸時間（秒） |

#### `LickResult`（dataclass）

`analyzer.update()` 每幀回傳的完整結果。

| 欄位 | 說明 |
|------|------|
| `current_zone` | 本幀接觸的區域（`BODY / FL / FR / HL / HR / NO_TARGET`） |
| `best_zone` | 截至目前累計時間最長的區域 |
| `body/fl/fr/hl/hr` | 各區域的 `ZoneStats`（hits + time_sec） |
| `face_state` | 頭部朝向標籤（`FRONT / LEFT / RIGHT / DOWN / UNKNOWN`） |
| `state_stability` | 多數決平滑中最高票標籤的佔比（0~1） |
| `dist_px` | 雙耳像素距離 |
| `dist_norm` | 雙耳距離正規化值（除以軀幹長） |
| `gaze_fwd / gaze_lat` | 前視/側視方向分量（供 Node-RED P3 面板使用） |
| `gaze_angle` | 頭部偏轉角度（度） |
| `valid` | 本幀骨架資料是否充分（False 時其他欄位為預設值） |

`to_payload()` 方法將 `LickResult` 序列化為可直接發布至 Node-RED 的 JSON 字典，`NaN/Inf` 浮點數自動轉為 `null`。

### 3.5 `contact_regions.py` — 幾何計算

#### `compute_geometry(keypoints, config) → dict`

根據本幀關節座標動態建構所有接觸目標區域。

**梯形建構流程（`_build_nose_trapezoid`）：**

1. 計算鼻→體中心的方向單位向量 `axis`
2. 以 `axis` 的垂直方向（`perp`）定義梯形左右邊界
3. 依 `NOSE_TRAP_TOP_W_RATIO`（頂邊窄）與 `NOSE_TRAP_BOT_W_RATIO`（底邊寬）計算四個頂點
4. 梯形厚度（深度）依 `NOSE_TRAP_THICKNESS_CM` 乘以像素/公分比例換算；若無法換算則使用 `NOSE_TRAP_THICKNESS_PX_FALLBACK`
5. 最終回傳 shape `(4, 2)` 的 float64 頂點陣列

**身體橢圓（`_build_body_ellipse`）：**

- 中心點：頸部與體中心的中點
- 長軸方向沿頸→體向量
- 長軸半徑 = 頸→體距離 × 0.55，短軸半徑 × 0.35

**肢體目標區域：**

- 每個肢體建立一個矩形條帶（limb strip）從體中心延伸至關節末端
- 同時建立末端圓形（paw circle）作為精細接觸判斷區

**回傳結構（dict）：**

```python
{
    "nose_trap_pts":       np.ndarray (4,2),   # 梯形頂點
    "body_center":         (x, y),
    "body_axis_unit":      (dx, dy),
    "region_rx":           float,               # 橢圓長軸像素半徑
    "region_ry":           float,               # 橢圓短軸像素半徑
    "limb_strip_targets":  [...],               # 肢體條帶資訊
    "limb_targets":        [...],               # 肢體末端圓資訊
}
```

#### `find_nearest_zone(nose_xy, geom) → (hit: bool, zone_label: str, nearest_label: str)`

測試鼻尖 `(x, y)` 是否落入任一接觸區域：

1. 優先測試各肢體末端圓（`cv2.pointPolygonTest`）
2. 其次測試體中心橢圓
3. 若均未命中，找出最近的目標中心並回傳為 `nearest_label`（用於視覺高亮）
4. `hit=True` 時 `zone_label` 為實際接觸區域；`hit=False` 時 `zone_label="NO_TARGET"`

### 3.6 `head_direction.py` — 頭部朝向推論

#### `compute_head_ear_angle(left_ear_xy, right_ear_xy, nose_xy) → dict`

計算雙耳相對於鼻尖的角度與方向分量。

- `gaze_fwd`：前視分量（雙耳中點到鼻尖的縱向距離）
- `gaze_lat`：側視分量（雙耳水平方向差）
- `gaze_angle`：偏轉角度（度）

#### `infer_face_state_cat_centric(ear_angle_data, config) → str`

以貓自身坐標系推論頭部朝向：

| 輸出標籤 | 條件 |
|---------|------|
| `FRONT` | `gaze_fwd` > 前視門檻（鼻尖在雙耳前方） |
| `LEFT` | `gaze_lat` < 側視門檻（頭轉向左） |
| `RIGHT` | `gaze_lat` > 側視門檻（頭轉向右） |
| `DOWN` | `gaze_fwd` < 低頭門檻（鼻尖低於耳朵） |
| `UNKNOWN` | 資料不足或落在模糊區間 |

#### `infer_face_state_user_rules(ear_angle_data, config) → str`

以人為定義規則（對稱性、比例）輔助推論，彌補貓坐標系在特殊姿勢下的失準。

#### `check_front_view_guard(dist_norm, config) → bool`

**前視守衛**：當雙耳距離正規化比例落在 `[FRONT_VIEW_EAR_RATIO_MIN, FRONT_VIEW_EAR_RATIO_MAX]` 時，表示攝影機正對貓臉，此時方向推論可靠度低，回傳 `True` 表示建議降低本幀置信權重。

#### `smooth_state(state, history_deque, window) → (smoothed_state: str, stability: float)`

維護長度 `window`（預設 30）的朝向標籤 deque，以**多數決**選出最穩定的狀態。

- `stability` = 最高票標籤的佔比（例如 `0.8` 表示 30 幀中有 24 幀為同一標籤）
- 高 stability 代表頭部朝向持續穩定，可提升舔舐部位判斷的置信度

---

### 3.7 `ear_distance.py` — 耳距與體型估算

#### `compute_ear_distance(keypoints, config) → dict`

| 回傳欄位 | 說明 |
|---------|------|
| `dist_px` | 左右耳關節的像素距離 |
| `dist_norm` | `dist_px / body_scale`（正規化，移除個體體型差異） |
| `valid` | 兩個耳朵關節的 confidence 均超過門檻時為 `True` |
| `body_scale` | 軀幹長（胸部到臀部的像素距離），作為正規化基準 |
| `body_ear_ratio` | `dist_px / body_scale`（同 `dist_norm`，供前視守衛使用） |

**用途：**

- `dist_norm` 越接近 1.0（雙耳距 ≈ 體長）表示正面朝向攝影機
- `front_view_guard` 使用 `body_ear_ratio` 判斷是否為正臉視角

---

### 3.8 `overlay.py` — 視覺化覆蓋層

#### `draw_all_overlays(frame, geom, trap_pts, hit, zone_label, nearest_label, nose_xy, frame_idx, show)`

主入口，依序繪製：

1. **`_draw_body_region()`** — 繪製體中心橢圓、肢體條帶、爪子圓（使用半透明合成 `cv2.addWeighted`）
   - 接觸中的區域以亮色高亮，其他區域以暗色顯示
   - 爪子圓採較高透明度（alpha=0.72）確保顏色鮮明可辨

2. **`_draw_nose_trapezoid()`** — 繪製鼻部接觸梯形
   - 接觸時：綠色填充 + 脈衝邊框動畫（`pulse = 0.5 + 0.5 * sin(frame_idx * 0.28)`）
   - 未接觸時：藍色半透明填充
   - 接觸時在鼻尖位置繪製擴散漣漪圈（ripple）
   - 旁側顯示「`CONTACT -> FL`」或「`NOSE CONTACT ZONE`」文字標籤

**肢體配色方案（BGR）：**

| 肢體 | 接觸時 | 非接觸時 | 顏色描述 |
|------|--------|---------|---------|
| FL（前左） | 橘色 | 深橘 | `(0, 165, 255)` |
| FR（前右） | 萊姆綠 | 深綠 | `(0, 255, 100)` |
| HL（後左） | 洋紅 | 深紫 | `(220, 50, 200)` |
| HR（後右） | 黃色 | 深黃 | `(255, 230, 0)` |

## 四、資料流整合示意

```
攝影機幀
    │
    ▼
YOLO-Pose → keypoints (17, 3) [x, y, conf]
    │
    ▼
LickStagePlugin.process(frame, keypoints, frame_idx, fps)
    │
    ├─ LickAnalyzer.update()
    │       │
    │       ├─ EMA 平滑 keypoints
    │       ├─ compute_geometry()      → geom dict + trap_pts
    │       ├─ find_nearest_zone()     → hit, zone_label
    │       ├─ compute_head_ear_angle() → gaze 向量
    │       ├─ compute_ear_distance()   → dist_norm, body_ear_ratio
    │       ├─ infer_face_state_*()    → face_state
    │       ├─ check_front_view_guard() → is_front_view
    │       ├─ smooth_state()          → smoothed_state, stability
    │       ├─ 累計 ZoneStats
    │       └─ draw_all_overlays()     → 修改 frame in-place
    │
    └─ LickResult（結構化輸出）
            │
            ├─ to_payload() → JSON
            └─ NodeRedPublisher.publish()  → MQTT / HTTP
```

## 五、梯形接觸區域設計說明

### 為什麼使用梯形（而非圓形或矩形）？

貓咪舔舐時，鼻尖到接觸點的距離並非固定，會隨頸部伸展而變化。梯形設計的優點：

- **方向性**：梯形沿鼻→體中心方向延伸，自然對齊貓咪的舔舐方向
- **漸寬設計**：靠近鼻尖端較窄（`TOP_W_RATIO=0.10`），遠端較寬（`BOT_W_RATIO=0.20`），容許舔舐目標的位置誤差隨距離增加
- **動態跟隨**：每幀根據當前關節位置重建，自動適應體型與姿勢變化

### 幾何參數視覺化

```
        鼻尖 (Nose)
          ▲
    ┌─────┼─────┐  ← 頂邊寬 = 體長 × TOP_W_RATIO (0.10)
    │     │     │
    │  梯形│     │  ← 深度 = NOSE_TRAP_THICKNESS_CM × px_per_cm
    │  接觸│     │
    └───────────┘  ← 底邊寬 = 體長 × BOT_W_RATIO (0.20)
          ▼
      體中心方向
```

### 接觸區域優先順序

當多個區域同時可能接觸時，以下優先順序決定 `zone_label`：

1. 肢體末端圓（FL / FR / HL / HR）— 精確判斷
2. 體中心橢圓（BODY）— 廣域判斷
3. 無命中（NO_TARGET）

## 六、擴充與調整建議

|------|------------|
| 調整梯形大小 | `config.py` → `NOSE_TRAP_*` 參數 |

| 新增 Node-RED 欄位 | `models.py` → `LickResult.to_payload()` |
| 更換視覺化配色 | `overlay.py` → `_PAW_COLORS` 字典 |
