> 本文件已依實際原始碼（2026-07-02）全面校對重寫。舊版文件描述的是較早期的設計草稿（class 名稱、方法簽名、狀態標籤、config 參數幾乎全數與現況不符），本次為完整改寫，而非局部修正。
>
> **2026-07-13 補充修正**：新增「一、設計思路」章節中 ST-GCN 行為門檻閘門的說明（此為本次更新最關鍵的行為變更——舊版文件明確寫錯，見下方修正）、`config.py` 的梯形彎曲自適應參數（`KP_MID_BACK`／`CURVATURE_*`）、`contact_regions.py` 的 `mid_back_dist_pct`/`curvature_boost` 計算，並更新「四、資料流整合示意」圖以反映閘門位置。

## 一、目的與需求背景

### 問題描述

ST-GCN 動作分類器輸出的行為標籤（如 `lick`）僅能判斷「貓咪正在舔舐」，無法進一步區分舔舐的**目標身體部位**（舔身體、舔前腳、舔後腳等）。然而在行為分析或醫學評估場景中，舔舐部位的差異具有重要意義：

- 持續舔舐特定肢體可能代表傷口、皮膚病或寄生蟲
- 舔舐頻率與分佈可作為焦慮或疼痛行為的量化指標
- 頭部朝向與接觸區域的組合判斷能過濾視角干擾造成的誤判

### 設計思路：二階段處理

```
第一階段：ST-GCN 行為分類（processors/frame_processor.py）
  ↓ 本幀已確認行為 = lick 且 confidence ≥ STGCN_BEHAVIOR_LABEL_CONFIDENCE_THRESHOLD ？
  │
  ├─ 是 → 餵入本幀真實關鍵點 (raw_kpts, kpt_conf)
  └─ 否 → 餵入 (None, None)（視同「貓不在畫面」）
        ↓
第二階段：lick_stage 精細化分析（獨立插件，逐幀呼叫 update()，但輸入內容由第一階段結果決定）
  ├── 計算耳距與身體尺度，判斷正面視角保護（front-view guard）
  ├── 若非正面視角：建立鼻部接觸梯形與身體/肢體目標區域
  ├── 測試鼻尖梯形是否與任一區域相交，取「距離最近」者為接觸區域
  ├── 推斷頭部朝向（FACING_CAMERA / FRONT_LEFT / FRONT_RIGHT / BACK_VIEW / FRONT_VIEW / NO_CAT / UNKNOWN）
  └── 輸出結構化 LickResult（含區域統計、朝向、穩定度、bout 統計）
```

> **本插件（`update()`）本身不含任何「是否為 lick」的判斷邏輯**——`LickAnalyzer.analyze()` 收到什麼就處理什麼，完全不知道、也不需要知道 ST-GCN 的分類結果。真正的「只在 lick 時才啟動」是由呼叫端 **`processors/frame_processor.py`** 決定：只有當本幀已確認的 ST-GCN 行為等於 `lick` 且信心值達到 `BehaviorTrackingConfig.STGCN_BEHAVIOR_LABEL_CONFIDENCE_THRESHOLD` 門檻時，才把本幀真實關鍵點傳給所有已註冊 plugin（包含 `LickStagePlugin` 與 `ext_body_zones/ExtBodyZonePlugin`）；其餘所有情況（`walk`/`scratch`/`shake`/`stop`/低信心/貓不在畫面）一律傳 `(None, None)`，讓 plugin 走既有的「無貓」重置路徑（`analyzer.py` 的 `_handle_no_cat()`）。
>
> 這是本模組**這次修訂前後最關鍵的行為差異**：舊版文件（含更早的版本）曾寫「本插件實際上每幀都會執行，並非只在 ST-GCN 判斷出 lick 標籤時才啟動」——這句話目前已不成立，設計選擇如下：
>
> - **不會污染統計**：`(None, None)` 對應 `zone_label="NO_TARGET"`，`LickStatistics.update()`（見 3.5）查表後直接跳過，不會把非舔舐時段的時間算進任何部位的累計時間；`models.py`（3.4）的百分比分母也只由已累計的部位時間組成，與非舔舐時段長短完全無關。`ext_body_zones/plugin.py` 的 `_run()` 在 `kpts is None` 時同樣提早 return、不寫入任何 zone 的 `time_sec`，兩個 plugin 的統計都是自我封閉的。
> - **判斷用的是即時（非顯示平滑）行為值**：`frame_processor.py` 另有一套 `_display_behavior_id`（hysteresis 平滑，只用於畫面 overlay/Node-RED 顯示防閃爍），但餵給 lick_stage 的閘門用的是未經平滑的即時 `self._last_behavior_id`/`self._last_confidence`——與 tracker/CSV/segment_logger 等其他「資料記錄」路徑共用同一份訊號，全系統判斷基準一致。
> - **代價（刻意接受）**：每次從非 lick 切回 lick，`_ema_kpts`/`_prev_trap_perp`/`_prev_trap_dir` 等跨幀平滑狀態會重置為初始值（與「貓咪離開畫面又回來」時完全相同的重置路徑），梯形方向需要重新收斂數幀才穩定；這是刻意接受的短暫視覺過渡，不代表資料錯誤。
> - 「二階段」一詞現在同時成立於兩個層次：**分析目的**的層次（粗分類 → 精細化定位）與**呼叫時機**的層次（第一階段須先確認才觸發第二階段的實際運算）。
>
> 詳細的閘門判斷式與逐行正確性驗證見 `processors/frame_processor.py` 中 `is_lick_behavior` 的計算，以及「四、資料流整合示意」。

## 二、模組架構與責任劃分

```
plugins/lick_stage/
├── manager.py          ← 對外公開介面（Plugin Facade）
├── analyzer.py         ← 逐幀協調器（Stateful Orchestrator）
├── config.py            ← 所有可調參數集中管理（LickConfig）
├── models.py             ← 資料結構定義（ZoneStats、LickResult）
├── statistics.py         ← 區域時間/次數/bout 統計累計（LickStatistics）
├── publisher.py           ← 非阻塞 Node-RED HTTP 發布（NodeRedPublisher）
├── contact_regions.py    ← 幾何計算：梯形建構 + 區域相交判斷
├── head_direction.py      ← 頭部朝向推論（含平滑與前視守衛）
├── ear_distance.py        ← 耳距計算與身體比例估算
├── overlay.py             ← 視覺化覆蓋層繪製（OpenCV）
├── __init__.py            ← 對外只匯出 LickStagePlugin
└── ext_body_zones/        ← 獨立的姊妹插件（見下方說明），與本模組無程式碼依賴關係
```

| 檔案 | 職責 | 是否有狀態 |
|------|------|-----------|
| `manager.py` | 接收原始骨架資料，呼叫 analyzer，發布 Node-RED，另提供獨立的 overlay 繪製介面 | 否（委派給 analyzer / publisher） |
| `analyzer.py` | 維護 EMA 平滑狀態、朝向歷史、上一幀 overlay 快取；委派統計給 `LickStatistics` | **是** |
| `config.py` | 定義關鍵點索引、梯形幾何、門檻值等所有超參數（`LickConfig` 純類別屬性，非 dataclass） | 否（靜態資料類別） |
| `models.py` | 定義 `ZoneStats`、`LickResult` 輸出結構與 JSON 序列化 | 否（純資料） |
| `statistics.py` | 累計各區域命中次數、累計時間、bout（連續舔舐段落）次數與時長 | **是** |
| `publisher.py` | 以 `ThreadPoolExecutor` 非阻塞方式 POST 結果至 Node-RED | 否（內部執行緒池，但無業務狀態） |
| `contact_regions.py` | 依鼻尖與軀幹關鍵點動態建構梯形與各目標區域，並找出「距離最近」的接觸區域 | 否（純函式） |
| `head_direction.py` | 以耳朵/軀幹相對位置推算頭部朝向，包含多數決平滑與前視守衛 | 否（純函式） |
| `ear_distance.py` | 計算雙耳像素距離，並以軀幹長正規化 | 否（純函式） |
| `overlay.py` | 將梯形、身體橢圓、肢體條帶、爪子圓繪製到畫面上 | 否（副作用函式） |

**未列入插件邏輯的檔案：**

- `1_自動抓取.py`：與 lick_stage 分析邏輯完全無關，是一支從 Pixabay/Pexels API 抓取貓咪行為影片供資料收集用的爬蟲腳本，僅是放置在此目錄下，並非插件架構的一部分。

**與 `ext_body_zones/` 插件的關係：** 該子目錄是**完全獨立的姊妹插件**（`ExtBodyZonePlugin`），提供另一種 7 區身體分區分類（HEAD / NECK_CHEST / SIDE_BACK / ABDOMEN / FORELIMB / HINDLIMB / TAIL），沿用同一套 17 點關鍵點順序，但擁有自己獨立的 `config.py` / `models.py` / `regions.py` / `output.py`，不呼叫、不匯入本模組的任何函式，於 `server/routes.py` 中以 `_try_register_ext_body_zone()` 單獨註冊，可獨立移除而不影響 `LickStagePlugin`。

> **兩者共用同一道 ST-GCN 閘門**：程式碼層面兩個 plugin 彼此獨立，但 `frame_processor.py` 是用同一個 `for _plugin in self._plugins: ...` 迴圈、同一個 `is_lick_behavior` 判斷結果去呼叫兩者的 `update()`（見上方「設計思路：二階段處理」）。因此雖然沒有程式碼依賴，兩者在「只在 lick 時才收到真實資料」這件事上是同步的。

## 三、各子模組詳細說明

### 3.1 `manager.py` — 對外公開介面

#### 類別：`LickStagePlugin`

系統其他元件只需與此類別互動，內部實作細節完全封裝。整合契約（見原始碼 docstring）：每個處理幀呼叫一次 `update(kpts, kpt_conf)`；`update()` 內部所有例外都會被捕捉並以 DEBUG 等級記錄，絕不會讓主系統崩潰；整個插件目錄可直接刪除而不影響主系統（`routes.py` 以 `try/except ImportError` 方式註冊）。

```python
class LickStagePlugin:
    def __init__(self, nodered_url: str = LickConfig.NODERED_URL): ...
    def update(self, kpts, kpt_conf) -> None: ...
    def draw_overlay(self, frame, frame_idx: int = 0, show: bool = True) -> None: ...
    def close(self) -> None: ...
```

| 方法 | 功能說明 |
|------|---------|
| `__init__` | 建立內部 `LickAnalyzer`；若 `nodered_url` 非空則建立 `NodeRedPublisher`，否則不推送 |
| `update(kpts, kpt_conf)` | `kpts` 為 `(17, 2)` 座標陣列（或 `None` 代表無貓），`kpt_conf` 為 `(17,)` 信心值陣列；內部呼叫 `analyzer.analyze()` 取得 `LickResult` 並（若有 publisher）發布至 Node-RED；**不回傳值、不接收 frame** |
| `draw_overlay(frame, frame_idx, show)` | 獨立的畫面疊圖方法，讀取 analyzer 上一幀快取的幾何資訊繪製到 `frame`；與 `update()` 完全解耦，可在不同時機呼叫 |
| `close()` | 關閉 `NodeRedPublisher` 的執行緒池（`wait=False`） |

> **與舊版文件的差異**：舊版描述 `__init__(config, publisher)` / `process(frame, keypoints, ...) → LickResult` / `reset()`，這些簽名在目前程式碼中均不存在。目前是 `__init__(nodered_url)` / `update(kpts, kpt_conf) → None` / `draw_overlay(...)` / `close()`。

### 3.2 `analyzer.py` — 逐幀協調器

#### 類別：`LickAnalyzer`

持有所有跨幀狀態，每幀呼叫一次 `analyze(kpts, kpt_conf, frame_idx, elapsed_sec, dt_sec) -> LickResult`。

**內部狀態（`__init__`）：**

| 屬性 | 說明 |
|------|------|
| `_stats` | `LickStatistics` 實例（並非文件舊版所稱的 `_zone_stats` 字典） |
| `_state_history` | `deque(maxlen=STATE_SMOOTH_WINDOW)`，朝向標籤歷史（用於多數決平滑） |
| `_ema_kpts` | 上一幀 EMA 平滑後的關鍵點（並非 `_prev_kps`） |
| `last_trap_pts` / `last_hit` / `last_zone_label` / `last_nose_xy` / `last_geom` / `last_nearest_label` | 供 `draw_overlay()` 讀取的上一幀 overlay 快取 |

> 沒有獨立的 `_frame_count` 屬性在 `LickAnalyzer` 上——幀計數實際上維護在 `LickStagePlugin` 內。

**實際處理流程（`analyze()` → `_handle_cat()`，非固定 12 步驟，而是含分支的流程）：**

```
analyze(kpts, kpt_conf, frame_idx, elapsed_sec, dt_sec)
  │
  ├─ 若 kpts/kpt_conf 為 None → _handle_no_cat()：記錄 NO_CAT 狀態，回傳空結果
  │     （此分支現在有兩種觸發來源，analyzer 內部無法、也不需要分辨兩者：
  │      ① YOLO 真的沒偵測到貓；② frame_processor.py 判定本幀 ST-GCN
  │      行為不是已確認的 lick，因而刻意傳入 (None, None)。兩者都應該
  │      視為「本幀不產生舔舐部位資料」，走同一條重置/NO_TARGET 路徑
  │      在語意上是正確的——見「一、設計思路：二階段處理」）
  │
  └─ _handle_cat()：
        1. EMA 平滑關鍵點（EMA_ALPHA 預設 1.0 = 不平滑，直接略過）
        2. compute_ear_distance()  → dist_px, dist_norm, valid, body_ear_ratio
        3. check_front_view_guard(kpt_conf, dist_px, body_ear_ratio) → front_guard
        4. compute_head_ear_angle() → angle_deg（供後續 user_rules 使用）
        │
        ├─ 若 front_guard=True（正臉朝鏡頭，幾何判斷不可靠）：
        │     → 直接跳過 compute_geometry / find_nearest_zone
        │     → 狀態設為 BACK_VIEW（若鼻子信心低）或 FRONT_VIEW，穩定度強制 = 1.0
        │     → zone_label = "NO_TARGET"
        │
        └─ 若 front_guard=False：
              → compute_geometry() 建立梯形與各身體目標區域
              → infer_face_state_cat_centric() 取得貓體坐標系初判狀態
              → infer_face_state_user_rules() 以角度/距離規則覆寫（優先權更高）
              → 若規則有觸發且狀態屬於 FRONT 系列，穩定度強制 = 1.0；否則呼叫 smooth_state() 多數決平滑
              → find_nearest_zone() 在所有候選區域中取「距離最近者」為接觸區域
              → 更新 overlay 快取（last_trap_pts 等）
        │
        6. self._stats.update(zone_label, dt_sec) → 委派 LickStatistics 累計
        7. 回傳 _build_result(...) 組成的 LickResult
```

> **overlay 繪製不在此流程中**：`analyzer.analyze()` 完全不觸碰畫面、不接收 `frame` 參數；繪圖是由 `LickStagePlugin.draw_overlay()` 另外呼叫 `overlay.draw_all_overlays()` 完成，讀取的是 analyzer 快取下來的上一幀幾何資訊。

### 3.3 `config.py` — 參數集中管理

#### 類別：`LickConfig`

所有閾值與幾何參數均在此定義為類別屬性（非 `dataclass`），修改時只需改這一個檔案。

**關鍵點索引（沿用專案通用的 17 點 YOLO-Pose 骨架編號，與 `utils/constants.py` 一致）：**

| 參數 | 值 | 對應部位 |
|------|----|---------|
| `KP_NOSE` | 0 | 鼻尖 |
| `KP_LEFT_EAR` | 1 | 左耳 |
| `KP_RIGHT_EAR` | 2 | 右耳 |
| `KP_CHEST` | 3 | 前胸 |
| `KP_MID_BACK` | 4 | 背部中心（標記於背部最高點，**非** chest-hip 連線中點——兩者只在脊椎打直時重合，蜷曲/拱背時會明顯偏離，這正是下方梯形彎曲自適應的量測基礎） |
| `KP_HIP` | 5 | 髖部 |
| `LIMB_SEGMENTS` | `[("FL",6,7), ("FR",8,9), ("HL",10,11), ("HR",12,13)]` | 四肢（膝關節索引, 腳掌索引） |

> lick_stage 收到的是**完整未截斷的 17 點陣列**（`frame_processor.py` 在 ST-GCN 專用的 14 點截斷處理**之前**就把原始 `kpts`/`kpt_conf` 傳給所有已註冊插件），因此這裡的索引與專案整體的 0–16 解剖對照表一致，並非誤用。

**信心值門檻：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `EAR_CONF_THRESHOLD` | 0.5 | 耳朵關鍵點最低信心值 |
| `NOSE_CONF_THRESHOLD` | 0.5 | 鼻子關鍵點最低信心值 |
| `LIMB_CONF_THRESHOLD` | 0.10 | 四肢關鍵點最低信心值 |
| `BODY_KP_CONF` | 0.5 | 胸部/髖部信心值門檻（計算身體尺度用） |

**頭部朝向狀態標籤（實際字串值）：**

| 常數 | 字串值 | 說明 |
|------|--------|------|
| `STATE_UNKNOWN` | `"UNKNOWN"` | 無法判斷 |
| `STATE_FRONT` | `"FACING_CAMERA"` | 正面朝向鏡頭 |
| `STATE_FRONT_LEFT` | `"FRONT_LEFT"` | 偏左的正面 |
| `STATE_FRONT_RIGHT` | `"FRONT_RIGHT"` | 偏右的正面 |
| `STATE_BACK` | `"BACK_VIEW"` | 背對鏡頭 |
| `STATE_FRONT_VIEW` | `"FRONT_VIEW"` | 前視保護觸發（暫停舔舐判定） |
| `STATE_NO_CAT` | `"NO_CAT"` | 畫面中無貓 |

> 沒有獨立的 `LEFT` / `RIGHT` / `DOWN` 狀態——「偏左/偏右」一律是 `FRONT_LEFT`/`FRONT_RIGHT`（正面朝向下的細分），也沒有低頭（`DOWN`）狀態。

**貓咪視角凝視方向門檻：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `CAT_FRONT_FORWARD_MIN` | 0.08 | 判定「朝前」的前向分量最小值 |
| `CAT_BACK_FORWARD_MIN` | 0.10 | 判定「背後」的前向分量最小值 |
| `CAT_LR_MARGIN` | 0.06 | 左右偏移容許誤差 |
| `CAT_LR_SIGN` | 1.0 | 左右方向符號（鏡像時可設 -1） |

**使用者規則門檻：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `FRONT_CAMERA_ANGLE_MIN_DEG` | 45.0 | 耳鼻夾角超過此值判定為面朝鏡頭 |
| `FRONT_CAMERA_NORM_MIN` | 0.30 | 耳間距標準化值輔助門檻 |
| `BACK_CAMERA_NOSE_CONF_MAX` | 0.5 | 背對判定的鼻子信心值上限 |
| `BACK_CAMERA_DIST_MIN_PX` | 3.0 | 背對判定所需耳間距最小像素值 |
| `BACK_VIEW_REQUIRE_LOW_NOSE` | `True` | 背對判定是否要求鼻子信心值必須低於門檻 |

**正面視圖保護（front-view guard）：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `FRONT_VIEW_GUARD_ENABLED` | `True` | 是否啟用 |
| `FRONT_VIEW_BODY_EAR_RATIO_MAX` | 0.75 | `body_scale / dist_px` **單側上限**觸發保護（並非雙側 min/max 區間） |

> 舊版文件聲稱有 `FRONT_VIEW_EAR_RATIO_MIN=0.85` / `FRONT_VIEW_EAR_RATIO_MAX=1.15` 雙門檻，該欄位不存在；實際只有單一上限 `FRONT_VIEW_BODY_EAR_RATIO_MAX=0.75`，且比例方向是 `body_scale/dist_px`（耳距越小、比值越大越可能觸發），與舊版描述的比例方向相反。

**身體區域幾何參數：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `BODY_ELLIPSE_W_RATIO` | 0.65 | 身體橢圓寬度比例（相對身體長） |
| `BODY_ELLIPSE_H_RATIO` | 0.27 | 身體橢圓高度比例 |
| `HEAD_RAY_LENGTH_RATIO` | 1.60 | 頭部方向射線長度比例（相對耳間距） |
| `HEAD_RAY_MIN_PX` | 60.0 | 頭部射線最短像素長度 |

**鼻子接觸梯形幾何參數：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `NOSE_TRAP_THICKNESS_CM` | 2.2 | 梯形深度（公分，換算像素） |
| `NOSE_TRAP_THICKNESS_SCALE` | 1.0 | 深度縮放係數 |
| `NOSE_TRAP_TOP_W_RATIO` | 0.10 | 梯形頂寬比例 |
| `NOSE_TRAP_BOT_W_RATIO` | 0.20 | 梯形底寬比例 |
| `NOSE_TRAP_W_SCALE` | 1.15 | 寬度整體放大係數 |
| `CAT_BODY_LENGTH_CM` | 40.0 | 標準身體長度，用於像素/公分換算 |

> 舊版文件聲稱的 `NOSE_TRAP_THICKNESS_PX_FALLBACK=22`（像素後備值）不存在；實際上是以 `CONTACT_BODY_LEN_MIN_PX`/`MAX_PX` 夾鉗身體長度，再換算像素/公分比例，沒有獨立的像素後備常數。

**梯形彎曲自適應（2026-07-13 新增）：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `CURVATURE_PCT_MIN` | 0.0 | `mid_back_dist_pct` 下限：脊椎接近打直 |
| `CURVATURE_PCT_MAX` | 30.0 | `mid_back_dist_pct` 上限：明顯拱背/蜷曲 |
| `CURVATURE_BOOST_MIN` | 0.85 | 對應 `CURVATURE_PCT_MIN` 時的梯形縮放倍率（打直時縮小） |
| `CURVATURE_BOOST_MAX` | 1.20 | 對應 `CURVATURE_PCT_MAX` 時的梯形縮放倍率（拱背時放大） |

> **問題背景**：`body_len` 是 chest-hip 兩點的**直線**距離。貓咪拱背/蜷曲理毛時，脊椎會明顯偏離這條直線，此時直線距離會低估「真實」身體尺度，導致鼻部接觸梯形在最需要精準判定的姿勢下反而系統性偏小。`mid_back_dist_pct`（`mid_back` 到 chest-hip 中點的正規化距離百分比，由 `contact_regions.py` 計算，見 3.7）就是這個彎曲程度的量測值；`CURVATURE_*` 四個參數定義把它線性內插成一個縮放倍率的區間，只套用在梯形尺寸上（不影響身體橢圓/四肢區域——那些量測的是「當下真實」幾何，不該被姿態推測放大縮小）。

**接觸幾何尺寸夾鉗：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `CONTACT_BODY_LEN_MIN_PX` / `MAX_PX` | 300.0 / 650.0 | 身體長度像素夾鉗範圍 |
| `LIMB_CONTACT_SCALE` | 1.0 | 四肢接觸區域整體縮放 |
| `LIMB_PAW_CIRCLE_R_RATIO` | 0.04 | 腳尖圓圈半徑比例 |
| `LIMB_STRIP_HW_RATIO` | 0.055 | 四肢長條寬度比例 |
| `LIMB_STRIP_EDGE_GAP` | 0.0 | 長條與腳尖圓間距 |

**其他：**

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `STATE_SMOOTH_WINDOW` | 30 | 朝向多數決平滑窗口（幀數） |
| `EMA_ALPHA` | **1.0**（不平滑） | 關鍵點 EMA 係數；舊版文件誤稱名為 `KP_EMA_ALPHA` 且預設 0.7 |
| `ZONE_NO_TARGET` / `ZONE_BODY` | `"NO_TARGET"` / `"BODY_CENTER"` | 區域標籤常數 |
| `ELLIPSE_SAMPLES` | 40 | 橢圓邊界取樣點數 |

| `NODERED_TIMEOUT` | 0.3 秒 | HTTP 逾時（避免卡住主迴圈） |

### 3.4 `models.py` — 資料結構定義

#### `ZoneStats`（dataclass）

| 欄位 | 型別 | 說明 |
|------|------|------|
| `hits` | `int` | 接觸事件次數（從其他區域/無接觸轉入該區域計一次） |
| `time_sec` | `float` | 累計接觸時間（秒） |

#### `LickResult`（dataclass）

| 欄位 | 說明 |
|------|------|
| `current_zone` | 本幀接觸的區域（`BODY_CENTER / FL / FR / HL / HR / NO_TARGET`） |
| `best_zone` | 截至目前累計時間最長的區域（由 `LickStatistics.best_zone()` 決定） |
| `body/fl/fr/hl/hr` | 各區域的 `ZoneStats` |
| `face_state` | 頭部朝向標籤（見 3.3 狀態表） |
| `state_stability` | 多數決平滑最高票標籤佔比（0~1），或規則直接命中時強制為 1.0 |
| `valid` | 本幀骨架資料是否充分 |
| `frame` / `time_sec` | 幀序號 / 累計經過秒數 |
| `dist_px` / `dist_norm` | 雙耳像素距離 / 正規化值 |
| `gaze_fwd` / `gaze_lat` / `gaze_angle` | 貓體坐標系下的前向/側向分量與偏轉角度（僅供 Node-RED P3 面板使用） |

`to_payload()` 將 `LickResult` 攤平序列化為 Node-RED JSON 字典（例如 `body.time_sec` → `body_time`、`body.hits` → `body_hits`），`NaN/Inf` 浮點數（`dist_px/dist_norm/gaze_fwd/gaze_lat/gaze_angle`）自動轉為 `null`。

### 3.7 `contact_regions.py` — 幾何計算

#### `compute_geometry(kpts, kpt_conf) -> Optional[dict]`

根據本幀鼻、胸、髖（以及可用時的雙耳）關鍵點動態建構所有接觸目標區域。若鼻/胸/髖任一信心值不足或胸髖距過小則回傳 `None`。

**回傳字典鍵值：**

```python
{
    "nose": ..., "ear_center": ...,
    "body_center": ..., "body_normal": ..., "body_axis_unit": ..., "body_len": ...,
    "mid_back_dist_pct": ..., "curvature_boost": ...,  # 2026-07-13 新增，見下方說明
    "region_rx": ..., "region_ry": ...,          # 身體橢圓半徑
    "nose_contact_trapezoid": ...,                 # (4,2) 梯形頂點（注意鍵名，非 nose_trap_pts）
    "limb_targets": [...],                          # 肢體末端圓
    "limb_strip_targets": [...],                    # 肢體條帶
    "trap_perp": ..., "trap_dir": ...,              # 未經跨幀穩定的原始方向向量，供 LickAnalyzer 平滑後重建梯形
    "trap_top_half": ..., "trap_bot_half": ..., "trap_height": ...,  # 已套用 curvature_boost 的梯形尺寸
}
```

#### `_curvature_size_boost(mid_back_dist_pct: float) -> float`（2026-07-13 新增，純函式）

依 `mid_back_dist_pct` 線性內插出梯形縮放倍率（區間定義見 3.3 的 `CURVATURE_*` 參數）：輸入為 `NaN`（`mid_back` 信心不足）或內插區間長度為 0 時回傳 `1.0`（不縮放，等同未量測時的既有行為）；否則依 `CURVATURE_PCT_MIN/MAX` 正規化並夾鉗到 `[0,1]`，再內插到 `[CURVATURE_BOOST_MIN, CURVATURE_BOOST_MAX]`。

`compute_geometry()` 內的計算順序：先以 `mid_back` 與 `body_center`（chest-hip 中點）的正規化距離算出 `mid_back_dist_pct`（`mid_back` 信心值需 `> EAR_CONF_THRESHOLD`，否則為 `NaN`），再呼叫 `_curvature_size_boost()` 得到 `curvature_boost`，最後把它**乘進** `trap_height`／`trap_top_half`／`trap_bot_half` 三個梯形尺寸算式——只影響鼻部接觸梯形，不影響 `region_rx`/`region_ry`（身體橢圓）或 `limb_targets`/`limb_strip_targets`（四肢區域）。

#### `find_nearest_zone(target_geom) -> Tuple[str, float, bool]`

回傳 `(zone_label, distance, hit)`，而非舊版文件所述的 `(hit, zone_label, nearest_label)`。演算法**並非固定優先順序**（肢體優先→身體→無命中），而是：

1. 蒐集所有與鼻部梯形**相交**的候選區域（身體橢圓、各肢體腳掌圓、各肢體條帶）
2. 對每個候選計算鼻尖到該區域中心線/中心的距離
3. 在所有候選中取**距離最小者**作為最終 `zone_label`
4. 若無任何候選相交，回傳 `(NO_TARGET, nan, False)`

即「距離最近者勝出」，不是「肢體圓 > 身體橢圓 > 無命中」的階層式優先權。

## 五、梯形接觸區域設計說明

### 為什麼使用梯形（而非圓形或矩形）？

貓咪舔舐時，鼻尖到接觸點的距離並非固定，會隨頸部伸展而變化。梯形設計的優點：

- **方向性**：梯形沿鼻→體中心方向延伸，自然對齊貓咪的舔舐方向
- **漸寬設計**：靠近鼻尖端較窄（`NOSE_TRAP_TOP_W_RATIO=0.10`），遠端較寬（`NOSE_TRAP_BOT_W_RATIO=0.20`），容許舔舐目標的位置誤差隨距離增加
- **動態跟隨**：每幀根據當前關節位置重建，自動適應體型與姿勢變化

### 接觸區域判定邏輯（並非固定優先順序）

`find_nearest_zone()` 會蒐集鼻部梯形與**所有**候選區域（身體橢圓、四個肢體腳掌圓、四個肢體條帶）中「有相交」的項目，再從中選出**鼻尖距離最近**的一個作為最終接觸區域；若都沒有相交則回傳 `NO_TARGET`。因此不存在「肢體圓一定優先於身體橢圓」的階層規則——只是實務上肢體圓通常較小、較貼近鼻尖，距離自然容易勝出。
